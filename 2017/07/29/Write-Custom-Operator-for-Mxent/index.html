Posted on 2017-07-29

Implement ROIAlign Operation in Mxnet

Recently, I’m trying to implement ROIAlign operation (reference to mask rcnn) in mxent. I want to write down some keypoints here. Note my version is not official and I copy a lot of codes from github ([ to do: add references]). Anyway, what we focus here is not the specific implementation but some required steps for general cases.

Official Guidances

  1. How to create new operation
  2. Contribute to community: there are links to test cases and examples. Some files that I found useful are:
    1. test_operator.py
    2. test_utils.py
    3. custom_softmax.py
  3. Operator In Mxnet
  4. Source files for operator: Specifically, we may want to take a look at files in contrib folder.
  5. Mshadow: a lightweight CPU/GPU Matrix/Tensor Template Library in C++/CUDA. Just use it as a library when needed.

Specifically, we need to look at the implementation of ROIPoolingOp: roi_pooling-inl.h, roi_pooling.cc, roi_pooling.cu.

GPU Implementation

We focus on GPU implementation here.

Headers and enumerations

  • logging.h: you will find something like CHECK_LT(x, y) and LOG_INFO, which are handy for logging (see google logging).

  • parameter.h: I guess it is used for defining and parsing fields for our operators.

  • operator.h: Our custom operator has to extend Operator class.

  • mshadow_op.h: ? There are matrix operations using mshadow templates. All the operations are in mxnet::op::mshdow_op namespace.

  • operator_common.h: ? .This header includes utility functions operator can use

The following codes are enclosed in mxnet::op namespace. Sincerely speaking, I’m not familiar with namespace. But generally, I think namespace has two effects:

  1. The things you defined are in this namespace.
  2. When use those things outside the namespace, you have to use :: operation.

We actually use those enumerations as indics for input and outputs, e.g. input[roialign::kData], which returns input[0], are actually data input for this operator.

Operator parameters

We define the parameters that are addtional to those blob-like data here using the tools provided in the parameter.h.

Operator operations

Then we define the ROIAlignOp. Its constructor should accept pre-defined Parameter structure as input. explicit prohibits the implicit conversion.

We define virtual Forward function here. The real implementation is located at the final call to ROIAlignForward. Before that call, certain preparetaion are done, including checking, obtaining tensors from in_data and out_data and initialization of tensors. Initialization here requires functions of mashdow, e.g. max_idx_x = -1.0f; set all elements to -1.0 in the matrix. That’s why using namespace mashdow; is required. CHECK_EQs are typical functions in logging system.

Backward here are quite similar to Forward. It also need to do checking, obtain tensors and initialize them. In our case, the initialization depends on the request (to do: I’m not sure req is the abbreviation of this. req[i]`` could be *kNullOp*, *kWriteTo*, *kWriteInplace*, *kAddTo* ... ?). Note grad_in = 0.0f; set all gradients to 0. ROIAlignBackwardAcc undertakes the real task.

An important observation here is that we don’t require all the out_data, out_grad, in_data. For system to recycle memory effectively, we specify these dependencies with the DeclareBackwardDependency interface in OperatorProperty.

About req

I didn’t read the whole document but I guess this term is determined by the engine for the computational graph. For example, if a node bifurcate into two branches, then its gradients need to be accumulated from two sources. Therefore the type of req would be kAddTo in this case. In the normal setting, e.g. sequential connections, kWriteTo should be default.

Operator property

Here is an important separation: * Operator: implementation interface * OperatorProperty: semantic interface

The OperatorProperty interface consists of following several functions according to the documentation. * Infer Shapes: Still, from the perspective of a system, this function is used to allocate memory. From the viewpoint of users, this function help us to check the shape of tensors. * Request Resources: See Operator In Mxnet. It helps to access to additional space for computation. * Backward Dependency: It specify the indispensible ingredients to compute gradients for input so that make it possible for system to recycle the unnecessary ones. * In-place Option: It helps to suggest system to reuse tensors for data and gradients for element-wise operations. See Operator In Mxnet. * Expose Operator to Python: there are several functions designed for python API, like initial function, ListArguments(), ListOutputs(). See Operator In Mxnet for a detailed list and explaination.
