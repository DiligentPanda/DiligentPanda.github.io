Faster R-CNN: Towards Real-Time Object Detection with Region Proposal Networks

Based on Fast R-CNN as detector, Faster R-CNN incorporate proposal generation stage into net by sharing conv features with detector and sliding a mini-subnet over the feat maps. The computation time for porposals is sharply reduced and the proposals’ quality is high.

  • Shaoqing Ren, Kaiming He, Ross Girshick, Jian Sun
  • Advances in Neural Information Processing Systems(NIPS), 2015
  • https://arxiv.org/abs/1506.01497
  • https://github.com/rbgirshick/py-faster-rcnn
  • https://github.com/ShaoqingRen/faster_rcnn

Abstract

  1. Introduce Region Proposal Network(RPN)
    • SPPNet, Fast R-CNN rely on external region proposal algorithms: slow.
    • RPN shares the fully convolutional features with the detection net.
      • cost-free region proposals.
      • predict object bounds and objectness scores at each position.
  2. Results
    • VGG-16
    • 73.2% voc2007; 70.4% voc2014: 300 proposals
    • 5fps

Introduction

  1. Sharing convolution cross proposals: R-CNN -> Fast R-CNN
    • less time
    • computational bottleneck lies in region proposal
  2. RPN share features with object detection network.
    • On top of shared feature map:
      • extract feature vector
      • predict objectness and regression for k proposals relative to k anchors of various scales and radios at a location

Apporach

Region Proposal Networks

  • Sliding window (size \(3*3\)) -> \(3*3\) conv -> two \(1*1\) conv
  • 9 anchors at one position: 3 scales, 3 aspect radios, 4k outputs for encoding coords, 2k scores. Total \(kHW\) anchors for conv feat map of size \(H*W\).
  • traslation invariant: address the problems of obj translation (because it is convolutional.): less paramters compared to MultiBox.

Leaning Region Proposal

  • Positive samples:
    • the highest IoU overlap with a gt bbox
    • IoU overlap with a gt bbox is higher than 0.7
  • Negative samples:
    • IoU overlaps with all gt bboxes are less than 0.3

  • Others: No contrib

  • Loss
    • \(*\) means gt. \(p_i^*\) is either 0 or 1 (gt labels: neg/pos).
    • \(t\) is a encoding of coords (\(x,y,h,w\), center coords and sizes) with respect to anchors.
    • reg loss is only activated for pos anchor.

  • Learnable weights: NOT SHARED for k anchors: one for each (scale, shape).

Optimization

  • randomly sample 256 rois for rpn loss in a mini-batch.
  • for sampled rois, pos:neg = 1:1, pad pos if less pos.
  • fix some paramters to conserve memory.
  • Sharing convolutional features
  • complex 4-step training algo

Implementation details

  • resize image: shorter side to 600
  • total strides of last conv feature in VGG/ZF : 16.
  • 3 scales: \(128^2,256^2,512^2\) (8,16,32 relative to the stride (in the feature map))
  • 3 aspect radios: \(1:1,1:2,2:1\)
  • \(1000*600\) image: \(60*40*9\) anchors
  • ignore boundary boxxes during training
  • hard to train.
  • 6k anchors left.
  • NOT in testing.
  • RPN NMS
  • IoU threshold: 0.7.
  • 2k proposals for training Fast R-CNN.
  • top-N (300 is good) ranked used for detection in test.
  • Those proposal are transformed by their predictions to rois as regions for feature extraction in Fast RCNN.

Other things

  1. Why it is called region-base cnn? In my understanding, this is because since its first generation, in the network, there are operations are region-based(or roi-wise). This is inherent in object detection tasks.

  2. alternative finetuning or train jointly?

  3. It is not clear a priori if learning Fast R-CNN while simultaneously changing the proposal mechanism will converge.

  4. We note that our algorithm allows the use of anchor boxes that are larger than the underlying receptive field when predicting large proposals. Such predictions are not impossible, one may still roughly infer the extent of an object if only the middle of the object is visible. I still think there might be improvement.

  5. It is noteworthy that the Recall-to-IoU metric is just loosely related to the ultimate detection accuracy. It is more appropriate to use this metric to diagnose the proposal method than to evaluate it.
