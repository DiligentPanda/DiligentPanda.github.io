Faster R-CNN: Towards Real-Time Object Detection with Region Proposal Networks

Based on Fast R-CNN as detector, Faster R-CNN incorporate proposal generation stage into net by sharing conv features with detector and sliding a mini-subnet over the feat maps. The computation time for porposals is sharply reduced and the proposals’ quality is high.

  • Shaoqing Ren, Kaiming He, Ross Girshick, Jian Sun
  • Advances in Neural Information Processing Systems(NIPS), 2015
  • https://arxiv.org/abs/1506.01497
  • https://github.com/rbgirshick/py-faster-rcnn
  • https://github.com/ShaoqingRen/faster_rcnn

Abstract

  1. Introduce Region Proposal Network(RPN)
    • SPPNet, Fast R-CNN rely on external region proposal algorithms: slow.
    • RPN shares the fully convolutional features with the detection net.
      • cost-free region proposals.
      • predict object bounds and objectness scores at each position.
  2. Results
    • VGG-16
    • 73.2% voc2007; 70.4% voc2014: 300 proposals
    • 5fps

Introduction

  1. Sharing convolution cross proposals: R-CNN -> Fast R-CNN
    • less time
    • computational bottleneck lies in region proposal
  2. RPN share features with object detection network.
    • On top of shared feature map:
      • extract feature vector
      • predict objectness and regression for k proposals relative to k anchors of various scales and radios at a location

Apporach

Region Proposal Networks

  • Sliding window (size \(3*3\)) -> \(3*3\) conv -> two \(1*1\) conv
  • 9 anchors at one position: 3 scales, 3 aspect radios, 4k outputs for encoding coords, 2k scores. Total \(kHW\) anchors for conv feat map of size \(H*W\).
  • traslation invariant: address the problems of obj translation (because it is convolutional.): less paramters compared to MultiBox.

Leaning Region Proposal

  • Positive samples:
    • the highest IoU overlap with a gt bbox
    • IoU overlap with a gt bbox is higher than 0.7
  • Negative samples:
    • IoU overlaps with all gt bboxes are less than 0.3

  • Others: No contrib

  • Loss
    • \(*\) means gt. \(p_i^*\) is either 0 or 1 (gt labels: neg/pos).
    • \(t\) is a encoding of coords (\(x,y,h,w\), center coords and sizes) with respect to anchors.
    • reg loss is only activated for pos anchor.

  • Learnable weights: NOT SHARED for k anchors: one for each (scale, shape).

Optimization

  • randomly sample 256 rois for rpn loss in a mini-batch.
  • for sampled rois, pos:neg = 1:1, pad pos if less pos.
  • fix some paramters to conserve memory.
  • Sharing convolutional features
  • complex 4-step training algo

Implementation details

  • resize image: shorter side to 600
  • total strides of last conv feature in VGG/ZF : 16.
  • 3 scales: \(128^2,256^2,512^2\) (8,16,32 relative to the stride (in the feature map))
  • 3 aspect radios: \(1:1,1:2,2:1\)
  • \(1000*600\) image: \(60*40*9\) anchors
  • ignore boundary boxxes during training
  • hard to train.
  • 6k anchors left.
  • NOT in testing.
  • RPN NMS
  • IoU threshold: 0.7.
  • 2k proposals for training Fast R-CNN.
  • top-N (300 is good) ranked used for detection in test.
  • Those proposal are transformed by their predictions to rois as regions for feature extraction in Fast RCNN.

Feature Pyramid Networks for Object Detection

  • Tsung-Yi Lin, Piotr Dollar, Ross Girshick, Kaiming He, Bharath Hariharan, Serge Belongie
  • CVPR 2017
  • https://arxiv.org/abs/1612.03144

Abstract

  • Feature Pyramid
  • exploit inherent multi-scale, pyramidal hierarchy of DCN: marginal extra cost.
  • bottom-up (CNN ifself) + top-down(high-level semantic information) + lateral connection(for information fusion)
  • detect objects of different scales.
  • Results
  • Use on Faster R-CNN > other single model
  • 5fps

Introduction

  • Feature Pyramid:
  • standard solution: image pyramid
  • scale-invariant: object scale change = shift its level in pyramid
  • applied in testing generally: mutli-scale images as input => all levels, including high resolution is semantically strong
  • limitation: inference time increase + end-to-end training is infeasible + inconsistency between training and testing inference
  • Exploit inherent CNN structure:
  • high resolution <-> low semantic
  • SSD doesn’t reuse high resolution feat maps, which is important for small object detection
  • FPN = bottom-up + top-down + lateral
    • proposals: AR +8%
    • detection: COCO AP +2.3%; VOC AP+3.8% > single-scale Faster R-CNN with ResNet
    • segmentation

Approach

Structure

  • Bottom-up
  • stage: layers that output feat maps of the same size.
  • last layers of stages to construct pyramid
  • Resnet: {C2,C3,C4,C5}=>stride {4,8,16,32}, C1 discarded.
  • Top-down path + lateral connection
  • add 1*1 conv on C5 => coarsest feat map for starting iter
  • {C2,C3,C4,C5} => merged feats
  • merged feats + conv \(3*3\) => final feats {P2,P3,P4,P5}:
    • conv 3*3
    • reduce aliasing effect of upsampling
    • output channels: 256-d, for shared classifier and regressor
    • no nonlinearity

Applications

FPN for RPN

  • original RPN:
  • head
  • anchor
  • on single-scale feature map
  • Adapted version:
  • on each level of pyramid
  • each level deal with one size of anchors.
    • {\(32^2,64^2,128^2,256^2,512^2\)} on {P2,P3,P4,P5,P6}
    • P6: Stride 2 upsampling of P5, only introduced for the \(512^2\) scale, not applied in Fast R-CNN detector
    • radios: still {\(1:1,1:2,2:1\)}
    • 15 kinds of anchors
    • Implementation: the same as original one, see Faster R-CNN
    • No need to explicitly assign GT bboxes to different levels

  • we have also evaluated the alternative without sharing parameters and observed similar accuracy. The good performance of sharing parameters indicates that all levels of our pyramid share similar semantic levels. This advantage is analogous to that of using a featurized image pyramid, where a common head classifier can be applied to features computed at any image scale.

FPN for Fast R-CNN

  • assign RoIs of diff scals to diff levels
  • roi (w,h) -> \(P_k\): \[k=\lfloor k_0+log_2(\sqrt{wh}/224) \rfloor\] > Here 224 is the canonical ImageNet pre-training size, and k0 is the target level on which an RoI with w * h = 224 * 224 should be mapped into.
  • \(k_0=4\)
  • Diff from original Fast R-CNN with ResNets
  • original
    • conv5 (9-layer sub-network) adopted as the head on top of conv4 feats
  • In this paper:
    • conv5 used for pyramid
    • share heads: 7*7 roi-pooling + 2 1024-d fc layers with relu + final classification and bbox regression layers.
    • lighter weight and faster

  • How does the naming system in mxnet work?

A network is composed of data blobs and layers. In the usual case, we can name layers at our will, e.g. name='FC_Layer' or it will get default name, e.g. fc1. The default name start counting at 1. It can be read from [sym].name.

Blobs are associated with layers, generally including input, output and weights. Because input/output data blob would be shared by multiple layers, the name of input data blob is specified/overwriten by the lower layer (the name of output data blob).

The output blob would be named as [layer_name]_[output_name], output_name is by default output[id] (possibably multiple outputs). We can get output_name by .list_outputs() method.

The paramter blob would be named as [layer_name]_[default_paramter_name], e.g. fc1_weight and fc1_bias. These rules also apply to our self-defined layers. If we provide parameters by a Variable, the name would be overwriten by the name of the Variable. This is consistent with the case when we input other data blobs into this layer. We can read parameter_name from .list_arguments(). But it is still not so straightforward, because .list_arguments() list all the arguments that is required in the whole computation graph rather than this node.

So, how about gradients of those blob? Their names seems purely defined as [layer_name]_backward_[default_blob_name], such as fc1_backward_weight, fc1_backward_bias, fc1_backward_data, even if input data blob actually has name data_img for example. Anyway, I really think this is confusing.

  • Note that data/label name provided by your data iter should match the ones required by your network.

Posted on 2017-08-10 | In paper/cv/detection

R-FCN: Object Detection via Region-based Fully Convolutional Networks

  • Jifeng Dai, Yi Li, Kaiming He, and Jian Sun
  • Conference on Neural Information Processing Systems (NIPS), 2016
  • https://arxiv.org/abs/1605.06409
  • https://github.com/daijifeng001/r-fcn
  • https://github.com/YuwenXiong/py-R-FCN

Abstract

This paper propose a fully convolutional structure for object detection, which removes the heavy roi-wise operations after roi-pooling. To compensate translation invariance which is original addressed by those roi-wise operations, position-sensitive score maps are introduced. This model saves a lot of time by removing roi-wise headers and sharing computations in fully convolutional way.

  1. Fully Convolution and Position-Sensitive Score Maps for Object Detection

    • Previous region based networks auch as Fast/Faster R-CNN apply a costly per-region subnetworks(those headers for classification and regression) hundreds of times. (Note after roi-pooling, even though we can implement headers in the convolutional way, the computation is no longer shared. Overlaps of rois are repeatedly computed.)

    • The region-based detector in this paper is fully convolutional, and allow all the computation shared on the entire image. This paper proposes position-sensitive score maps to address the dilemma between translation invariance in image classification and traslation variance in object detection. (By translation, we mean an object move relative to its bbox.)

  2. Results:
    • Resnet-101
    • 83.6% map on PASCAL VOC 2007. 82.0% on 2012.
    • 170ms/img in inference: 2.5-20 times faster then Faster R-CNN

Introduction

  1. RoI pooling layer divides previous prevalent deep object detectors into two subnets:
    • A shared convolutional one for feature extraction
    • A RoI-wise subnet don’t share computation

    resulted from old classification system.(final spatial pooling layer -> roi pooling layer)

  2. The state-of-art classification system is fully convolutional, like ResNets. In order to use fully convolutional backbone, we have to solve the aforementioned dilemma. Inserting RoI pooling (a region-specific operation) is a solution but it introduce heavy headers(those headers lead to translation variance). See Table 1.
    • TODO Review Fast R-CNN
    • Examples:
      • Translation of objects should reflect the change of overlap between gt object and bbox.
      • The left hands appering in the right of bbox should be the hands of others.
  3. Structure Comparison
    • fully convolution (output feature maps) -> roi pooling -> roi-wise opertaion (fc or conv)
    • fully convolution (output position-sensitive score maps) -> position-sensitive poooling -> without weighted layer

Approach

  • RPN & R-FCN share features
  • Backbone: ResNet-101 with feature reduction to 1024 channels.
  • ps score maps: \(k^2(C+1)\) channels, \(k^2\) is corespoing to \(k^2\) bin.
  • ps roi pooling: for class c, bin location x,y, selectively pool the corespondent channel. Aggregate scores from \(k^2\) channels.
  • Average \(k^2\) scores as final voting results.
  • Similar class agnostic bbox regresssion.
  • OHEM: online hard example mining.
  • hole algorithm.

Other Things

  1. The inference process is easy to understand. How about the training part? Specifically, how do final score maps learn to be position sensitive? Through end-to-end learning, the successive selective pooling shepherds the convolution layer to learn specialized position-sensitive score maps. This position sensitive score maps may be slightly different from the one in fcis. It only shows the potential a point may belong to a certain part of bbox. So a point didn’t belong to an object but belong to another object near this object may be evaluated as high probability. Even tough proposals may not overlap with the gt bbox in a perfect way (So, some one may lead to false learning of locations, e.g. left-up score map shows high value on the center of an object. Notice gt bboxes are always used in training as addtional proposals), the high probability region marked by those proposals is still reasonable if we think all proposals follows a gaussian like distribution. (The most centered bbox (the gt bbox) always gets the best score.)

Anyway the keypoints here we should understand is how end-to-end learning plays a critical role in this process and forces the layer to learn what we want.

A great thing about machine learning is that if you give the model flexility to learn critical information, it will always imporve. And if you give the model restrictions, it will learn things more easily.

Actually, in the second step of Faster R-CNN, the computation time is propotional to the number of propsals from the previous RPN subnet, typically 300 bboxes([300,2048(1024),h,w],to check Faster R-CNN paper), which is a really big cost.

Cython for NumPy

Cython

  • A compiler

Cython is an optimizing static compiler for both the Python and the extended Cython.

  • A language

Cython is a language that makes writing C extensions for the Python language as easy as Python itself.

Speed-Up

In the later discussion, we will achieve speed-up from the following aspects. * static data type * no negative indices * no boundary check * eliminates overhead of for-loop * eliminates overhead of fuction calls: specify type of parameters and return values * efficient indexing * …

Anyway, the main ideas here are:

  • Whatever is convenient in Python may be slow in execution. We convert those part into explicit coding in C to speed up the process. (Even though always a C file is generated for .pyx file, but runing of some codes need python interpreter, which makes execution slow.)
  • Those functionalities in Python are usually more generic. So we have to provide addtional information. If the compiler lacks of some information, the codes still runs in Python way rather than C. (For newbies like me, we don’t know how much information is sufficient for cython’s optimization.)
  • In another direction, we want to make use of many useful tools already implementd in Pyhon. For instance, We want to exploit Numpy, which provides convenient API of Multi-dimensional arrays. However, note that most of time-comsuming operations of this library are implemented in C fundamentally, which means when we call those functions, we would gain little speed-up.

Steps of Using Cython

  1. write .pyx source file
  2. run Cython compiler: generate C file
  3. run C compiler: generate a compiled library(.so, a shared library)
  4. import the libaray into other modules.

It is very IMPORTANT to check whether we obtain expected elegant C codes in .c file.

Compile Libaray by setup.py

Syntax

  1. First of all, we could write any python codes. When it is possible to be converted to C implementation, it would speed up.

  2. cdef to add types for variables:

For efficient indexing, we also specify the data type and the number of dimensions for ndarray.

Note that it can only be used at the top indentation level.

  1. cdef can also be used to define functions: it is used for Cython functions that are intended to be pure ‘C’ functions. Types of parameters and return value should be specified. However, this function can not be called in Python. We are writing Python codes, but evetually it is C code.

Functions defined by def can be called direcly from Python codes. We can specify types for parameters of the function. (Return value as local variable can also be specified with a type.) However, it incurs a high overhead.

Besides, there is cpdef, which is both.
